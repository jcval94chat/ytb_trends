Análisis técnico integral del repositorio ytb_trends

Subtítulo: Pipeline batch en Python que toma históricos de Google Sheets, calcula métricas de tendencias y vuelve a publicar resultados en otras hojas de Google, incorporando extracción de Google Trends por país.

Tipo: Pipeline ETL/analítica batch Lenguajes: Python + YAML Runtime: Python 3.9 (CI) / 3.12.7 (README) Complejidad: Media Confianza: Alta (repo pequeño, código legible)

1) Resumen ejecutivo

El repositorio automatiza una cadena de procesamiento para detectar y puntuar keywords de YouTube/Trends por país. El flujo principal se ejecuta con google_trends_data.py en GitHub Actions, leyendo datasets históricos desde carpetas de Google Drive (cada archivo es una Google Sheet), preprocesándolos con métricas estadísticas (máximos, medianas, scoring diario/semanal/mensual), consultando Google Trends (pytrends) para keywords seleccionadas y escribiendo salidas a distintas hojas de cálculo. Existe también un workflow de keepalive para evitar inactividad prolongada del repositorio.

3) Flujo completo del dato

  1. Trigger: GitHub Actions programa google_trends_workflow.yml por cron y manual.
  2. Carga de secretos: CI decodifica credenciales y exporta variables de entorno para carpetas/hojas.
  3. Entrada principal: google_trends_data.py valida secretos críticos y corta ejecución si faltan.
  4. Lectura histórica: get_sheets_data_from_folder lista archivos de Drive, filtra por timestamp en nombre y concatena dataframes.
  5. Transformación analítica: preprocesar_keys deriva métricas diarias, filtra ventana de 60 días, estima punto de corte con histograma y extrae best/worst.
  6. Selección de keywords: se toma subconjunto permitido desde df_key_words_ (umbral por cuantíl sobre mean_interest).
  7. Extracción Trends: print_trends consulta pytrends por país/periodo en chunks de 5 términos.
  8. Salida: resultados se escriben en varias hojas de cálculo (keywords, métricas, best/worst).
  9. Errores/logs: manejo de excepciones con logging + stacktrace; en faltantes críticos usa exit(1).
Diagrama de flujo: 
GitHub Cron/Manual
   -> google_trends_data.py
      -> get_sheets_data_from_folder(folder 1) -> df_key_words
      -> get_sheets_data_from_folder(folder 2) -> combined_df_keys

      -> print_trends(keywords_permitidos, MX/US)
      -> save_dataframe_to_gsheet + upload_dataframe_to_google_sheet

5) Entrypoints reales

  • python google_trends_data.py: entrypoint de negocio.
  • .github/workflows/google_trends_workflow.yml: scheduler y disparador manual.
  • .github/workflows/keepalive.yml: job de mantenimiento.
  • Inferencia: no hay API HTTP, CLI formal ni workers separados; el modelo operativo es batch script.

6) Interfaces y movimiento de datos

  • Google Drive API: lista archivos por folder (query parents).
  • Google Sheets API (gspread): lectura worksheet 0 y escritura de tablas completas.
  • Pytrends: build_payload + interest_over_time + trending_searches.
  • Formato dominante: DataFrame tabular en memoria; persistencia en Google Sheets.
  • Contrato implícito: columnas esperadas como date, interest, keyword, country.

7) Ejecución y despliegue

  • Local: instalar requirements.txt y definir variables de entorno/secrets.
  • CI/CD real: GitHub Actions en Ubuntu instala dependencias y corre el script principal.
  • Testing: no existen tests automáticos en repo (ni carpetas tests/ ni comandos en workflow).
  • Infra: no hay Docker, Compose, Terraform, Kubernetes.

Variables críticas detectadas: SECRET_FOLDER_ID, SECRET_FOLDER_ID_DF, SECRET_CREDS_FILE, GOOGLE_SHEETS_CREDS_BASE64, SPREADSHEET_ID_TRENDS, SPREADSHEET_ID_KEYWORDS, SPREADSHEET_ID_KW, SPREADSHEET_ID_BBDD.

8) Dependencias importantes

  • pandas, numpy: transformación y scoring de series temporales.
  • pytrends: consumo no oficial de Google Trends.
  • gspread, google-api-python-client, oauth2client, google-auth*: integración con Sheets/Drive.
  • matplotlib, seaborn: visualización auxiliar (no integrada al pipeline CI).

9) Caso práctico narrado (E2E)

Un jueves a las 00:00 UTC se dispara el workflow principal. El job descarga el repo, instala dependencias y decodifica credentials.json. El script carga dos carpetas de Drive: una para keywords históricas y otra para dataset base. Con estos datos calcula métricas por día y define un conjunto de keywords candidatas por país (MX/US). Luego consulta Google Trends en ventanas now 7-d y today 1-m, obteniendo series de interés. Finalmente, publica: (1) tabla de interés de keywords por país y tiempo, (2) tablas de best/worst y métricas analíticas en hojas separadas. Si faltan secrets clave o falla autenticación, termina con error y logs detallados.

10) Riesgos y puntos ciegos

Hecho confirmado: no hay suite de pruebas; la calidad depende de ejecución directa en CI y datos reales.
Hecho confirmado: discrepancia de runtime (README pide Python 3.12.7; workflow usa 3.9), riesgo de incompatibilidades futuras.
Hecho confirmado: alta dependencia de nombres de archivos con timestamp en Drive (parse_timestamp_from_name), sensible a convenciones manuales.
Hecho confirmado: el pipeline asume columnas específicas en sheets; no hay validación de esquema formal antes del preproceso.
Inferencia: el filtrado anti-fútbol extenso puede introducir falsos negativos por matching simple substring.
Inferencia: pytrends puede presentar rate limiting/intermitencia; hay manejo de excepción, pero no política explícita de reintentos exponenciales.
